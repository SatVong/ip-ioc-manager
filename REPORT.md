# Отчёт о разработке IP/IOC Manager

## История создания

Проект начинался как внутренний инструмент для управления списками IP-адресов и индикаторов компрометации (IOC) в рамках мониторинга угроз кибербезопасности так как спустя год ведение Exce, он себя изжил и расплодились десятки версий файлов с разным количеством записей что повлияло на актуализацию средств защиты. Проект запущен 10.01.2026 в виду не особо хорошим владением ИИ страница IP источники была написана собственно руками примерно за неделю потом проект был отложен на 3 месяца по причине огромной загруженности и сведения Excel в 1 файл. Добрался до deepseek процесс пошел быстрее.
Изначально была реализована только вкладка "IP-источники" на ванильном JavaScript с Express.js бэкендом. Потом у отдела появилась необходимость в IOC и Белых IP и постепенно возрастал функционал, но так как весь проект написан на ванильном JS возникли проблемы с реализацией некоторых функциональных частей в том числе с созданием тестов.

## Этапы разработки

### Этап 1: Базовая функциональность (старт)
- Создана структура проекта: backend (Node.js/Express) + frontend (HTML/CSS/JS)
- Развёрнута PostgreSQL с таблицами users, ip_records, user_logs
- Реализована JWT-авторизация с ролевой моделью
- CRUD для IP-записей с редактированием ячеек inline
- Система МСЭ-квадратиков (15 источников)

### Этап 2: Расширение функциональности
- Добавлена страница IOC-хешей с автоопределением кодировки (MD5/SHA1/SHA256/SHA512)
- Добавлена страница Белых IP
- Реализована серверная пагинация с фильтрацией и поиском
- Система прав пользователей (can_create, can_edit, can_delete, etc.)
- Импорт/экспорт CSV с валидацией
- Тёмная/светлая тема
- Механизм исключения записей с отслеживанием кто/когда
- Административные операции очистки таблиц

### Этап 3: Рефакторинг
- Монолитные JS-файлы (~1800 строк) разбиты на модули
- Создана структура: ui/, actions/, pagination/, validators/
- Каждый файл точки входа сокращён до ~70-80 строк +/-
- CSS вынесен в модульную структуру (15 файлов)
- Добавлен Swagger/OpenAPI для документирования API (31 эндпоинт)

### Этап 4: Контейнеризация
- Созданы Dockerfile для backend (Node.js Alpine) и frontend (Nginx Alpine)
- Docker Compose для оркестрации 3 контейнеров
- init.sql для автоматической инициализации БД
- Настроен nginx как reverse-прокси для API

## Ключевые проблемы и решения

### 1. Инцидент с рефакторингом (фатальный)
**Проблема:** 18.06.2026 При рефакторинге кода AI агентом через расширение CONTINUE в была повреждена структура frontend - удалилась часть index.html фрагменты CSS, JS (app.js) сайт был убит.
Такое поведение агента было связано скорее всего с тем что он не мог прочитать app.js который содержал ~1800 и при запросе к нему на рефакторинг он более 40 раз пытался его прочитать указывая:

**Решение:** Пошаговое восстановление из-за ведения чата с deepseek. Восстанавливал почти 20 версий изменений. Копирую код повторно с чата.
Из-за плохо владения git и vscode поплатился 4 днями восстановления. 

**Вывод:** При работе с AI инструментами критически важно иметь git репозиторий с коммитами после каждого успешного изменения что стал делать после такого случая. Сейчас не получается настроить AI агент в CONTINUE потратил на это сутки. Перешл на CLINE проблем не возникло. Пришлось делать все через част c deepseek.

### 2. Проблемы с импортами ES-модулей
**Проблема:** После разбиения на модули возникли циклические зависимости и неправильные пути импорта (`../ipExceptions.js` вместо `./ipExceptions.js`).

**Решение:** Поэтапная отладка через консоль браузера, исправление путей, вынесение общих функций в отдельные модули (ipExceptions.js).

### 3. Конфликт версий PostgreSQL в Docker
**Проблема:** На хосте была PostgreSQL 15, а Docker-контейнер использовал версию 16. Существующий том с данными вызвал ошибку несовместимости.

**Решение:** Полная очистка Docker volumes (`docker compose down -v`), создание чистого тома.

### 4. Ошибка подключения к БД в Docker
**Проблема:** Backend подключался к `localhost:5432`, но PostgreSQL был в другом контейнере так как сайт был на PM2.

**Причина:** В `db.js` были захардкожены параметры подключения вместо использования переменных окружения.

**Решение:** Переписан `db.js` на использование `process.env.*` и добавлен `.env` с правильным хостом `database`.

### 5. Сложности с тестированием на Vanilla JS
**Проблема:** Без фреймворка тестирование UI требовало бы JSDOM или браузерных тестов, что значительно усложнило бы процесс.

**Решение:** Принято решение отложить тесты до миграции на React (v2.0), сосредоточившись на ручном тестировании и проверке API через Swagger.

## Удачные решения

1. **Модульная структура frontend** - позволила легко поддерживать три почти идентичные страницы
2. **Docker-контейнеризация** - одной командой разворачивается всё окружение (Огромный минус в среде ЗКВС так как проблематично делать миграцию думаю над алгоритмом и рекомендациями полученными от коллег на занятиях в чате)
3. **Серверная пагинация** - высокая производительность даже с большими объёмами данных
4. **Swagger-документация** - упрощает интеграцию и тестирование API
5. **Система прав** - гибкая ролевая модель с тонкой настройкой разрешений
6. **Автоопределение хешей** - автоматически распознаёт MD5/SHA1/SHA256/SHA512 по длине

## Неудачные решения

1. **Хардкод параметров в db.js** - вызвал проблемы при Docker-развёртывании
2. **Прямые импорты ES-модулей без сборщика** - много HTTP-запросов при загрузке страницы

## Фатальные решения

1. **Изначально монолитный код** - привёл к необходимости масштабного рефакторинга из-за которого локальный AI положил весь код
2. **Отсутствие Git с самого начала** - первое знакомство с VCS, работа без репозитория увеличила риски
